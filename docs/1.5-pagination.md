# 1.5 Milestone — Pagination on Heavy Tools

Add `limit` / `offset` to the four tools that can return unbounded lists, and report `total` / `hasMore` / `nextOffset` so an LLM client can page without blowing its context.

Delete this doc when 1.5 ships.

---

## Motivation

Four tools currently return full lists:

- `find_references` — every use of a symbol across the project
- `find_symbol` — every matching declaration
- `analyze_imports` — `incoming` (importers of a file)
- `analyze_routes` — every route in the project

On a 500-component codebase, a single `find_references('Button')` can return 200+ hits across 80+ files. The MCP response is one text blob; an LLM reading it pays for every line. Pagination prevents context blowup and makes these tools usable on serious repos.

---

## API contract

### Input (added to each tool's `inputSchema`)

```ts
{
  limit?:  number;  // default 100, max 1000
  offset?: number;  // default 0
}
```

- `limit` ≤ 0 → reject with `Invalid arguments: limit must be > 0`
- `limit` > 1000 → clamp to 1000 and include a note
- `offset` < 0 → reject

### Output (added to each tool's response)

Top-level response object gains:

```ts
{
  total:       number;          // matches BEFORE pagination
  limit:       number;          // applied limit
  offset:      number;          // applied offset
  hasMore:     boolean;         // true when offset + matches.length < total
  nextOffset:  number | null;   // offset + limit when hasMore, else null
}
```

Existing array fields (`matches`, `references`, `incoming`, `routes`) hold the **sliced** page, not the full list. Consumers that want all results iterate with `nextOffset`.

## Scope per tool

### 1. `find_symbol` — [src/tools/find-symbol.ts](../src/tools/find-symbol.ts)

**Current response:**
```ts
{ name, kindFilter, matches: SymbolMatch[], total, filesScanned }
```

- `total` already exists but means "filtered matches count" (same as `matches.length`). Redefine: `total` = matches count before pagination. Keep `filesScanned` as-is.
- Slice `matches` with `limit`/`offset` after filtering.
- **Sort:** matches are already ordered by glob traversal + in-file order, which is deterministic enough. Explicitly sort by `(file, line)` before slicing so the order is stable across pages even if the glob implementation changes.

### 2. `find_references` — [src/tools/find-references.ts](../src/tools/find-references.ts)

**Current response shape** (verify in source): `{ symbol, file, references: Reference[], total, filesScanned }`.

- Same treatment as `find_symbol`. Sort by `(file, line, column)` before slicing.
- **Note for implementer:** `find_references` scans every file. Pagination only caps output size, not scan cost. If scan cost becomes the bottleneck, that's the AST-cache follow-up, out of scope here.

### 3. `analyze_imports` — [src/tools/analyze-imports.ts](../src/tools/analyze-imports.ts)

Only paginate `incoming` — it's the unbounded one (every file that imports this one). `outgoing` is bounded by a single file's import list and never grows past ~50.

**Current response:**
```ts
{ file, outgoing: {...} | null, incoming: Importer[] | null }
```

- Add pagination fields **at top level** alongside `file`. `total` applies to `incoming.length`.
- When `direction === 'outgoing'` (no incoming requested), pagination fields should still appear but with `total: 0, hasMore: false, nextOffset: null`. This keeps the response shape stable.
- Sort `incoming` by `(file, line)`.

### 4. `analyze_routes` — [src/tools/analyze-routes.ts](../src/tools/analyze-routes.ts)

**Current response:** `{ routes: Route[], ... summary counts ... }`.

- Paginate `routes`. Keep summary counts computed over the full set.
- Sort by `path` so pages are lexicographically ordered (deterministic across calls).

## Shared implementation

Add `src/tools/shared/pagination.ts` with:

```ts
export const DEFAULT_PAGE_LIMIT = 100;
export const MAX_PAGE_LIMIT = 1000;

export interface PageInfo {
  total: number;
  limit: number;
  offset: number;
  hasMore: boolean;
  nextOffset: number | null;
}

export function paginate<T>(
  items: T[],
  limit: number = DEFAULT_PAGE_LIMIT,
  offset: number = 0
): { items: T[]; page: PageInfo };

export function paginationArgsSchema(): z.ZodObject<{
  limit: z.ZodOptional<z.ZodNumber>;
  offset: z.ZodOptional<z.ZodNumber>;
}>;

export function paginationJsonSchema(): {
  limit: { type: 'number'; minimum: 1; maximum: 1000; default: 100 };
  offset: { type: 'number'; minimum: 0; default: 0 };
};
```

Each tool imports and composes — no duplicated validation logic.

## Tests

Per tool, add to the existing test file:

- **Default behavior** — call without pagination args on a 5-match fixture → all 5 returned, `total: 5`, `hasMore: false`, `nextOffset: null`.
- **Paged** — `limit: 2, offset: 0` → first 2, `hasMore: true`, `nextOffset: 2`.
- **Last page** — `offset: 4, limit: 2` on 5 matches → 1 item, `hasMore: false`, `nextOffset: null`.
- **Past end** — `offset: 10` on 5 matches → `[]`, `total: 5`, `hasMore: false`.
- **Stable order** — fetch full list, then fetch in 2-item pages, concatenate, assert equal.

Also add one **schema validation** test per tool: `limit: 0` and `limit: -1` → error; `offset: -1` → error.

Fixture should have ≥5 deterministically-ordered matches. [tests/tools/find-symbol.test.ts](../tests/tools/find-symbol.test.ts) already has a good fixture pattern — extend.

## Out of scope

- **AST cache** (Option A's other half). Pagination caps output, not scan cost. Cache is a separate milestone.
- **Streaming responses.** MCP text responses are single blobs. Streaming would need protocol-level work.
- **Paginating `analyze_components` / `analyze_hooks` / `analyze_features` / `analyze_patterns` / `analyze_i18n`.** These use the plugin architecture and their own summary-shaped output; pagination for them is a separate design problem. Plugin-based tools can be tackled as a follow-up once the shared helper proves out on the four self-contained tools above.
- **Cursor-based pagination.** Offset-based is simpler and sufficient for bounded result sets (no one's paging past 10k results). Cursors matter when the underlying set mutates mid-iteration; here the project is static within a session.

## Future hooks this unlocks

- **AST cache** becomes trivially useful — paged calls to the same tool hit the same files. A 2nd-call warm cache turns pagination from "network latency × N" into "filesystem latency once, N cheap slices."
- **Composite / chained tools** (e.g., "find every component whose props include `onClick`") — can iterate through pages without exceeding context on intermediate results.
- **Streaming** becomes a drop-in replacement for pagination once MCP supports it — consumers that already understand `hasMore` / `nextOffset` can just keep reading from a stream.
