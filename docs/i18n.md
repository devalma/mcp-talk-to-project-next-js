# i18n Detection

The `analyze_i18n` tool (and the `nextjs-i18n` CLI) flags untranslated user-facing strings in a React/Next.js project. It uses a **whitelist** approach with 7 specialized validators rather than a blacklist.

This doc covers **what it detects, why, and how** — in that order. If you just want to run it, see [tools.md `analyze_i18n`](tools.md#analyze_i18n).

---

## Why whitelist over blacklist

1. **Conservative** — only flags strings in contexts we're confident are user-facing.
2. **Predictable** — a developer can tell at a glance whether a string will be picked up.
3. **False-positive resistant** — technical strings (`"POST"`, `"flex"`, `"#fff"`) never accidentally end up in translation bundles.
4. **Maintainable** — a handful of explicit lists, easy to extend as new patterns appear.
5. **Fast** — list membership is cheaper than pattern-matching an open-ended set of technical idioms.

Start with the high-priority validators. When a real project has a legitimate user-facing string the extractor misses, extend the whitelist or add a validator rather than loosening the algorithm.

---

## The 7 validators

Priority signals how load-bearing a match is — high-priority matches are almost always translations, medium-priority ones depend on context. All validators use AST-level matching (Babel parser) plus content-quality filters (see below).

### High priority

#### 1. JSX text content — `jsx-text`

Direct text children of JSX elements. Always translatable.

```jsx
<h1>Welcome to our application</h1>
<p>Please enter your email address</p>
<span>Loading...</span>
```

AST node: `JSXText`.

#### 2. User-facing JSX attributes — `jsx-attribute`

Allow-listed HTML attribute names.

```jsx
<img alt="User profile picture" />
<button title="Click to save your changes" />
<input placeholder="Enter your email" />
<div aria-label="Close" />
<meta content="App description" />
```

**Attribute allow-list:** `alt`, `title`, `placeholder`, `label`, `aria-label`, `content` (on `<meta>`).

#### 3. User message variables — `variable-declaration`

Variables whose names signal user-facing intent.

```js
const welcomeMessage = "Welcome back!";
const errorText = "Something went wrong";
const successNotification = "Changes saved successfully";
const confirmLabel = "Yes, I agree";
```

**Name fragments:** `message`, `text`, `notification`, `alert`, `title`, `description`, `label`.

### Medium priority

#### 4. User-facing object properties — `object-property`

Object keys that signal user-facing values.

```js
const notifications = {
  success: "Data saved successfully!",
  error: "Failed to save data",
  warning: "Please check your input",
};

const menuItems = [
  { label: "Dashboard" },
  { label: "Settings" },
];
```

**Key allow-list:** `message`, `text`, `label`, `title`, `description`, `content`, `placeholder`, `tooltip`, `hint`, `help`, `error`, `success`, `warning`, `info`, `loading`, `errorMessage`, `successMessage`, `warningMessage`, `infoMessage`, plus common greetings/actions (`welcome`, `save`, `cancel`, `submit`, `confirm`, `continue`, `back`, `next`, …) and navigation fragments (`displayName`, `menuTitle`, `breadcrumb`).

#### 5. Form validation messages — `form-validation`

Validation rule strings — often in schema objects.

```js
const validation = {
  required: "This field is required",
  email: "Please enter a valid email",
  password: "Password must be at least 8 characters",
};
```

Combines the object-property logic with validation-specific context detection.

#### 6. Component props — `component-prop`

Props passed to React components (capitalized element names), distinct from HTML attributes.

```jsx
<Modal title="Confirm Delete" message="Are you sure?" confirmText="Delete" cancelText="Cancel" />
<Toast text="Item deleted successfully" />
<Tooltip content="Click to edit this field" />
```

**Prop allow-list:** `message`, `text`, `label`, `title`, `description`, `content`, `fallbackText`, `errorMessage`, `successMessage`, `warningMessage`, `infoMessage`, `confirmText`, `cancelText`, `submitText`, plus `tooltip`, `placeholder`, `helperText`, `hintText`, `statusText`, `actionText`, `buttonText`.

Element casing decides the validator: capitalized = component prop (V6), lowercase = HTML attribute (V2).

#### 7. Alert / prompt messages — `alert-message`

User-facing browser dialogs. `console.*` is explicitly filtered out.

```js
alert("Please save your work before leaving");
confirm("Do you want to delete this item?");
prompt("Please enter your name:");
```

**Flagged:** `alert`, `confirm`, `prompt`. **Ignored:** `console.log`, `console.debug`, `console.info`, `console.warn`, `console.error`, `console.trace`.

## Content-quality filters

Even inside a translatable context, a string must pass content filters:

**Accept if:**
- Length ≥ 3 with meaningful content
- Contains spaces, punctuation (`.!?:,`), mixed case, or `?`/`!` (confirmations/alerts)

**Reject if:**
- ≤ 2 characters
- Numbers/symbols only
- ALL_CAPS constants (`const VERSION = "1.2.3"`)
- CSS values (`"100px"`, `"flex"`, `"#ff0000"`, `"rgb(…)"`)
- Technical enum-like words (`"POST"`, `"button"`, `"primary"`)
- Boolean strings (`"true"`, `"false"`)

```ts
function isTranslatableContent(text: string): boolean {
  if (text.length < 2) return false;
  if (!text.trim()) return false;
  if (!/[a-zA-Z]/.test(text)) return false;

  const technical = [
    /^[A-Z_]+$/,                                // CONSTANTS
    /^[\d\.\-]+$/,                              // version numbers
    /^#[0-9a-f]{3,8}$/i,                        // color codes
    /^\d+(px|em|rem|vh|vw|%)$/,                 // CSS units
    /^(left|right|center|top|bottom|middle)$/,  // CSS positions
    /^(flex|grid|block|inline|none|auto)$/,     // CSS display values
  ];
  return !technical.some(p => p.test(text));
}
```

## What doesn't get flagged

The validators intentionally skip anything in these AST contexts:

- **Imports/exports** — `import React from 'react'`, `export { default } from './x'`
- **CSS/styling attributes** — `className`, `style`, `sx`
- **Technical HTML attributes** — `data-*`, `id`, `key`, `ref`, `type`, `role`
- **File paths and URLs** — anything starting with `/`, `./`, `../`, `@/`, or matching a URL pattern
- **Technical object keys** — `method`, `headers`, `contentType`, etc.
- **Environment variables and version constants**
- **Language keywords and library names**

## Decision tree

A quick "should this string get flagged?" algorithm:

```ts
function shouldTranslateString(path, text: string): boolean {
  const parent = path.parent;

  // Always
  if (parent?.type === 'JSXText') return true;

  // JSX attribute — allow-list
  if (parent?.type === 'JSXAttribute') {
    return TRANSLATABLE_ATTRIBUTES.includes(parent.name?.name);
  }

  // Object property — key allow-list
  if (parent?.type === 'ObjectProperty' || parent?.type === 'Property') {
    const key = parent.key?.name ?? parent.key?.value;
    return TRANSLATABLE_OBJECT_KEYS.includes(key);
  }

  // Template literal in JSX
  if (parent?.type === 'TemplateLiteral') {
    return isInJSXContext(path);
  }

  // Conservative default
  return false;
}
```

## Extending

The validator registry lives in `src/plugins/i18n-extractor/`. To add a new validator:

1. Implement the `BaseValidator` interface.
2. Register it in the validator registry.
3. Add patterns to the appropriate allow-list and content filter.

Keep new validators single-purpose and tiered by priority. Don't loosen the existing content-quality filters — extend the whitelist instead.
