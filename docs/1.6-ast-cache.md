# 1.6 Milestone — Session-scoped AST cache

Unify the five per-tool `safeParse` helpers behind one cached parser so repeated calls to the same file (especially the paged flows added in 1.5) stop re-parsing.

Delete this doc when 1.6 ships.

---

## Motivation

Five LLM-oriented tools have near-identical inline `safeParse(content, absFile)` functions:

- [src/tools/find-symbol.ts:369](../src/tools/find-symbol.ts#L369)
- [src/tools/find-references.ts:444](../src/tools/find-references.ts#L444)
- [src/tools/get-file-exports.ts:400](../src/tools/get-file-exports.ts#L400)
- [src/tools/get-component-props.ts:801](../src/tools/get-component-props.ts#L801)
- [src/tools/get-hook-signature.ts:271](../src/tools/get-hook-signature.ts#L271)

Each reads the file fresh and calls `@babel/parser`'s `parse()`. The plugin lists differ slightly (find-references has `topLevelAwait` + `allowReturnOutsideFunction`; the others don't), but the output AST shape is interchangeable for every consumer we have.

Two things make caching pay off now:

1. **Pagination (shipped in 1.5).** A `find_references('Button')` that returns 300 hits across 10 pages re-parses every file 10 times. Each page call walks the same glob, re-reads each file, re-parses it, then discards all but one page of results. A cache turns pages 2-10 into filesystem stats + map lookups.
2. **Session locality.** The natural LLM workflow — `find_symbol` → `get_component_props` → `find_references` → `analyze_imports` — hits many of the same files in sequence. Right now each tool pays the full parse cost.

Back-of-envelope: on a 500-file codebase, a paged `find_references` walks ~500 files × 10 pages = 5000 parses. With mtime-keyed caching, the first page costs 500 parses and subsequent pages cost 500 stat calls + map lookups. Expect page-2+ latency to drop from O(seconds) to O(tens of milliseconds).

---

## Non-goals (read these first)

Getting the scope right matters more than the implementation.

- **Not persistent.** In-memory only. No `.mcp-cache/` files. Persistent caching creates an invalidation surface (stale caches after `git pull`, across branches, after `npm install`) that's larger than the problem. A fresh MCP server process is a fresh cache, and that's fine.
- **Not a general memoizer.** Cache the parsed AST only. Don't cache `findReferences()` results, `findSymbol()` results, or any derived analysis — those depend on the _whole project_, and one changed file invalidates everything derived. Parsing is the per-file bottleneck; analysis on top of it is fast.
- **Not a traversal cache.** We're not caching `findImports(ast)` output either. Re-walking a cached AST is cheap; the parse is what hurts.
- **Not shared across tools via `context`.** Tempting to hang the cache off `ToolContext`, but the plugin-backed analyze tools don't use it and wouldn't benefit. A module-level singleton in `src/tools/shared/` is simpler and reaches everyone who imports it.

## API contract

Create [src/tools/shared/ast-cache.ts](../src/tools/shared/ast-cache.ts):

```ts
import type { File as BabelFile } from '@babel/types';

export interface ParsedSource {
  ast: BabelFile;
  content: string;
  absFile: string;
}

/**
 * Parse a file and return its AST, or null if the file is missing, empty,
 * or unparseable. Cached by (absFile, mtimeMs). If the file's mtime changes
 * between calls, the next call re-parses.
 *
 * The parser is configured with the superset of plugins used across the
 * LLM-oriented tools today — all existing callers' options are satisfied.
 */
export function parseFileCached(absFile: string): ParsedSource | null;

/** Drop all cached entries. For tests. */
export function clearAstCache(): void;

/** Cache stats for tests + potential future telemetry. */
export function getAstCacheStats(): {
  size: number;
  hits: number;
  misses: number;
  invalidations: number;
};
```

### Caching rules

- **Key:** `absFile` (after `path.resolve` normalization).
- **Invalidation:** `fs.statSync(absFile).mtimeMs`. On every `parseFileCached` call, stat the file first. If the cached entry's mtime matches, return it. Otherwise re-parse and replace.
- **Missing/unreadable files:** return `null`, don't cache. An `ENOENT` on a stat is a cache _miss_, not a cached `null`. This avoids serving stale nulls if a file is later created.
- **Unparseable files:** return `null` but _do_ cache the null alongside mtime, so syntactically-broken files don't re-incur parser overhead every page. Invalidate when mtime changes.
- **Bounded size:** cap at **500 entries** with simple LRU eviction (track `lastAccessed` timestamp, evict oldest when inserting past the cap). On a small project the cap is never hit; on a large one we avoid unbounded growth across a long session.

### Parser options

Unify on the superset of existing `safeParse` helpers:

```ts
{
  sourceType: 'module',
  allowImportExportEverywhere: true,
  allowReturnOutsideFunction: true,
  plugins: [
    'decorators-legacy',
    'dynamicImport',
    'exportDefaultFrom',
    'exportNamespaceFrom',
    'optionalChaining',
    'nullishCoalescingOperator',
    'topLevelAwait',
    ...(isTs ? ['typescript'] : []),
    ...(isJsx ? ['jsx'] : []),
  ],
}
```

`isTs` / `isJsx` by extension only — don't use the content-sniffing heuristic from [src/parsers/ast.ts:33-35](../src/parsers/ast.ts#L33-L35). That file's `content.includes('</')` check is a footgun; none of the five LLM tools rely on it.

**Verify before shipping:** run the full test suite against each tool after swapping. Any test that passed against the old plugin list and fails against the superset means we've enabled a plugin that changes AST shape for something a walker relies on.

## Scope per tool

For each of the five tools, replace the local `safeParse` + `readFileSafe` + `parse` chain with a single `parseFileCached` call. Delete the now-unused helpers.

Keep the existing error handling shape: if `parseFileCached` returns `null`, the tool's behavior should be identical to when `safeParse` previously returned `null` (skip the file, continue).

### 1. `find_symbol` — [src/tools/find-symbol.ts](../src/tools/find-symbol.ts)

- `findInFile` currently does `readFileSafe` → `safeParse` → traverse. Replace with `parseFileCached`.
- Delete [safeParse](../src/tools/find-symbol.ts#L369) and [readFileSafe](../src/tools/find-symbol.ts#L361).

### 2. `find_references` — [src/tools/find-references.ts](../src/tools/find-references.ts)

- `collectReferencesInFile` reads + parses every file. Replace both calls.
- Delete [safeParse](../src/tools/find-references.ts#L444) and [readFileSafe](../src/tools/find-references.ts#L440).
- **Highest expected speedup** — this tool walks every file in the project and is the primary motivator.

### 3. `get_file_exports` — [src/tools/get-file-exports.ts](../src/tools/get-file-exports.ts)

- Single-file tool, so cache benefit is lower per call. Still unify on the shared parser to delete the local helper.
- Delete the local `safeParse` + `readFileSafe`.

### 4. `get_component_props` — [src/tools/get-component-props.ts](../src/tools/get-component-props.ts)

- This one chases imports across files (`resolveImportedType`). Each hop parses a file. Common paths across a session: components that share a base type (e.g., `ButtonProps`) hit the same type definition file repeatedly.
- Replace both the entrypoint parse and the chase-parse. Delete the local helpers.

### 5. `get_hook_signature` — [src/tools/get-hook-signature.ts](../src/tools/get-hook-signature.ts)

- Single-file. Unify on the shared parser, delete the local helper.

### Not in scope

- **`analyze_imports.ts`** has its own `extractImports()` with an inline `parse()` call. It's also a good candidate but is a larger refactor — `extractImports` returns a rich `ParsedImport[]` that's a mix of AST walking and shape normalization. Do this separately after the five above ship cleanly. Leaving it for a follow-up keeps the 1.6 scope tight.
- **`analyze_routes.ts`** is regex-based, not AST-based. No parser to cache.
- **`src/parsers/ast.ts`** (`parseFile`) is only used by `src/plugins/manager.ts`. The plugin surface has its own caching via `PluginCache`. Leave untouched; do not try to unify plugin-side and tool-side parsing in 1.6.
- **`src/plugins/common/ast-utils.ts`** — same story. Plugins have a separate caching path.

## Shared implementation notes

### Invalidation race

`fs.statSync` + `fs.readFileSync` is not atomic. Between the stat and the read, a file could change. Two cases:

1. **Read succeeds with new content but we keyed on old mtime.** On the _next_ call, we'll see the new mtime and re-parse. Worst case: one stale result served. Acceptable — MCP tools are read-only, the user will see updated data on the next call.
2. **Read fails (file deleted between stat and read).** Catch, return `null`, don't cache. Next call will stat-miss and re-try cleanly.

### Concurrency

Node is single-threaded for user code. The MCP server handles one tool call at a time. No locking needed.

### Memory

500 entries × ~50-200 KB per AST ≈ 25-100 MB steady state. Acceptable for a dev-tools process. If this becomes a problem, drop the cap; the optimization target is latency, not memory.

### Testing hooks

Export `clearAstCache()` for test isolation. Every test file that touches a tool which now uses the cache should call `clearAstCache()` in `beforeEach` alongside the `mkdtempSync`.

Optionally export `getAstCacheStats()` so integration tests can _assert_ caching (e.g., "page 2 of find_references adds 0 cache misses over page 1"). Good for preventing silent regressions.

## Tests

New file: [tests/tools/shared/ast-cache.test.ts](../tests/tools/shared/ast-cache.test.ts).

Cover:

- **Hit path** — parse twice, second call returns the same AST object reference, `hits` counter increments.
- **Miss path** — first call increments `misses`, second increments `hits`.
- **Mtime invalidation** — parse, `fs.utimesSync` to bump mtime, parse again, `invalidations` increments and returned AST reflects new content.
- **Missing file** — returns `null`, nothing cached, subsequent create-then-call returns the new AST (not a stale `null`).
- **Unparseable file** — returns `null` and _does_ cache; second call is a hit (null served from cache).
- **LRU eviction at cap** — fill to 500 entries, insert 501st, verify oldest was evicted via `getAstCacheStats().size === 500`.
- **`clearAstCache()`** — resets size + counters.

Also add one **integration test** per paginated tool confirming that paging doesn't re-parse:

```ts
// tests/tools/find-references.cache.test.ts (or extend existing test file)
it('page 2 adds no new cache misses', async () => {
  clearAstCache();
  await findReferences(tmp, 'x', 'src/lib/target.ts', 2, 0);
  const missesAfterPage1 = getAstCacheStats().misses;
  await findReferences(tmp, 'x', 'src/lib/target.ts', 2, 2);
  expect(getAstCacheStats().misses).toBe(missesAfterPage1);
});
```

One test like this per tool is enough. If the tool ever grows a code path that parses something uncached, this test catches it.

## Out of scope

- **Persistent cache** — deliberate non-goal, see top.
- **Paginating `analyze_imports`** or extending its `extractImports` to use the shared cache — follow-up.
- **Shared cache for plugin-side parsing** (`src/plugins/manager.ts`, `src/plugins/common/ast-utils.ts`) — the plugin architecture has its own `PluginCache` layer; unifying is a bigger refactor and should not be bundled with this.
- **Cache observability tools** — no "show me cache stats" MCP tool. `getAstCacheStats()` is a test hook, not a user surface.
- **Tuning the LRU cap based on project size** — the 500 default is a fine starting point. Adjust if someone complains.

## Future hooks this unlocks

- **Composite tools.** A hypothetical `find_and_describe(name)` that calls `find_symbol` → `get_component_props` → `find_references` in one shot: the same files get parsed in multiple downstream tools. Cache makes composition essentially free.
- **Paginate `analyze_imports.incoming` with zero scan cost.** The incoming scan walks every file; after caching, pages 2+ only do resolver work on cached ASTs.
- **Shadow analysis in `find_references`** (noted in [next-steps.md](next-steps.md) §1). Needs `@babel/traverse` + scope tracking, which runs against the AST. Cheaper to add once the AST is cheap to get.
- **Possible 1.7: cross-tool result cache keyed on `(tool, args, project-fingerprint)`** with fingerprint = hash of all relevant file mtimes. Much bigger lift; mention only as a distant horizon.
