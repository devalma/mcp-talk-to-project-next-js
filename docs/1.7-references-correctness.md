# 1.7 Milestone — `find_references` correctness pass

Two known false-answer shapes in [`find_references`](../src/tools/find-references.ts), both of which were tolerable pre-1.6 because fixing them meant paying parsing costs we didn't want to pay. 1.6's AST cache makes parsing cheap. 1.7 cashes that in to make the answers correct.

Delete this doc when 1.7 ships.

---

## Motivation

`find_references(symbol, file)` has two documented defects. Neither is exotic; both erode trust when hit:

### 1. Barrel re-exports drop importers on the floor (false negatives)

Today: an importer that reaches the target through a barrel is not reported as a reference to the target. The barrel file itself is reported (existing code records a reference at its `export { X } from '…'` line), but the downstream consumer is invisible.

```ts
// src/lib/api.ts
export function fetchUser() {}

// src/index.ts                                 ← barrel
export { fetchUser } from './lib/api';

// src/app/page.tsx
import { fetchUser } from '..';                 ← WE MISS THIS
fetchUser();                                    ← AND THIS
```

`find_references('fetchUser', 'src/lib/api.ts')` returns the barrel line only. Rename `fetchUser`, trust the blast radius the tool reports, ship it — and `src/app/page.tsx` breaks at runtime.

Flagged in [`docs/next-steps.md`](next-steps.md) §1 ("`analyze_imports` / `find_references` — multi-hop re-exports"). Common in larger codebases with `index.ts` barrels; common in every Next.js component folder.

### 2. Shadowing produces false positives

Today: identifier matching is name-only — no scope analysis. A local binding that reuses the imported name gets counted.

```ts
import { fetchUser } from '../lib/api';
function greet(fetchUser: string) {              // shadows the import
  return fetchUser;                              ← reported as a reference, wrong
}
```

There is a test for this — [`tests/tools/find-references.test.ts:147`](../tests/tools/find-references.test.ts#L147) — that asserts only the import is reported, explicitly documenting the false positive as "acceptable for the MVP." Fixing it means the test flips to assert the wrong reference is no longer there.

Flagged in [`docs/next-steps.md`](next-steps.md) §1 ("`find_references` — shadow detection"). The fix was blocked on not wanting to introduce `@babel/traverse` or pay its parse cost on every call. Neither blocker applies now:

- `@babel/traverse` is already a transitive dependency and is already imported by the plugin-side utilities — see [`src/plugins/common/ast-utils.ts:6`](../src/plugins/common/ast-utils.ts#L6). The ESM default-export workaround (`const traverseFunction = typeof traverse === 'function' ? traverse : (traverse as any).default;`) is proven in-tree.
- The AST cache from 1.6 means a traversal is a map lookup + a walk, not a parse.

## Scope

**In:**
- Follow barrel re-export chains from the target, so importers of a barrel count as importers of the target.
- Use `@babel/traverse` scope bindings to drop shadowed name hits.
- Optional `via?: string[]` field on each `Reference` so LLMs can see which barrel(s) the reference passed through.

**Out:**
- `analyze_imports` barrel collapsing. Different semantics (`analyze_imports` answers "what does this file import?", which is literal). Mentioned in [`docs/next-steps.md`](next-steps.md) §1 as a separate lever; leave it.
- `find_symbol` scope awareness. `find_symbol` finds declarations, not uses. Shadowing is not a concern.
- Third-party barrel chasing. Only follow re-exports whose sources resolve to in-project files via the existing [`module-resolver`](../src/tools/shared/module-resolver.ts). `export * from 'lodash'` is not followed.
- `export * as ns from '…'` star-namespace re-exports. Rare; name mapping through a namespace is awkward. Record but don't follow — note in results.
- Default-export barrel chains with renaming. `export { default as Foo } from './x'` is a common enough pattern that we *should* handle it; call it out explicitly in tests.

## Design — barrel following

### The re-export index

A module-level helper, co-located with `find_references` (not in `shared/` — only this tool uses it for now):

```ts
// src/tools/find-references.ts (or a new file under src/tools/ if it grows)

interface BarrelEdge {
  /** File that owns the re-export. */
  from: string;             // abs path
  /** File the re-export resolves to. */
  to: string;               // abs path
  /** Name as exported by `from`, or '*' for star re-exports. */
  exportedName: string;
  /** Name as originally exported by `to`. `exportedName` differs when renamed. */
  originalName: string;
  /** Line of the re-export, for `via` attribution. */
  line: number;
  column: number;
}

function buildBarrelEdgesFor(
  targetAbsFile: string,
  projectPath: string,
  files: string[],
  aliases: TsconfigAlias[],
): BarrelEdge[];
```

The index is **target-directed**: we don't need a project-wide graph, only the set of files that transitively re-export from our target. Build it by:

1. Parsing each project file via `parseFileCached` (already scanned anyway — we were going to parse them for usage detection).
2. Recording every `ExportNamedDeclaration` with `source`, every `ExportAllDeclaration`, and their resolved destinations.
3. Doing a reverse BFS from `targetAbsFile`: file F is a barrel-of-target iff F re-exports (directly or transitively) from target.

Cycles: maintain a visited set during the BFS, drop revisits.

Worst-case cost: O(files) parses (cache makes them cheap after the first call in this process) + O(edges) traversal.

### Name mapping through barrel chains

For the target symbol `S` exported from `T`, the BFS produces a list of `(barrelFile, exposedName)` pairs — the *public* name each barrel gives the symbol:

- Direct re-export: `export { S } from 'T'` → `(barrel, 'S')`.
- Renamed: `export { S as X } from 'T'` → `(barrel, 'X')`.
- Star: `export * from 'T'` → `(barrel, 'S')` — preserves the name.
- Chained rename: barrel A renames S to X, barrel B re-exports `X` from A → `(B, 'X')`.
- Chained double-rename: B renames again → `(B, 'Y')`. Track the mapping per edge; don't assume the exposed name equals the original name.

Default-export re-exports — `export { default as Foo } from './x'` — treat as a named re-export with the exposed name `Foo`. If the consumer writes `import { Foo } from 'barrel'`, it counts. If the consumer writes `import Foo from 'x'`, that's the direct path and is handled by existing default-import matching.

Namespace star re-exports (`export * as ns from './x'`) — record an edge with a marker that disables following but enables a note in the barrel file itself. Out of scope for chased traversal in 1.7; add a TODO comment.

### Integration into `findMatchingImports`

Today `findMatchingImports` compares each importer's `source` to `absTarget` via `resolvesTo`. Extend this to compare against the set of barrel abs paths too, and when the match is through a barrel, use the barrel's *exposed* name to check the import specifier — not the original target name.

```ts
interface ResolvedTarget {
  absFile: string;
  exposedName: string;        // = symbol for direct, or the mapped name for barrel hops
  via: string[];              // [] for direct; barrel chain for indirect
}

// returned from the index: one direct + N barrel-indirect
function targetsFor(symbol: string, absTarget: string, ...): ResolvedTarget[]
```

Then for each importer, instead of `resolvesTo(source, …, absTarget, …)` → bool, do `findTargetFor(source, …, targets, …)` → `ResolvedTarget | null` and use the mapped `exposedName` when iterating specifiers.

`ResolvedImport` gains an optional `via?: string[]`. Each reference emitted downstream inherits it.

### Output shape change

```ts
export interface Reference {
  file: string;
  line: number;
  column: number;
  kind: RefKind;
  localName: string;
  /** When the reference reached the target through barrel re-exports,
   *  the chain of barrel files in order (nearest-to-importer first).
   *  Absent for direct imports. */
  via?: string[];
}
```

Purely additive. Existing consumers that ignore `via` see no change.

## Design — shadow detection

Swap the hand-rolled walker in `findLocalNameUsages` for a `@babel/traverse` visitor that consults `path.scope.getBinding(name)`.

```ts
import traverse from '@babel/traverse';
const traverseFunction = typeof traverse === 'function' ? traverse : (traverse as any).default;

function findLocalNameUsages(
  ast: BabelFile,
  localName: string,
  importSpecPath: any,            // to compare against the binding
): Array<{ line: number; column: number; kind: RefKind }> {
  const results = [];
  traverseFunction(ast, {
    Identifier(path) {
      if (path.node.name !== localName) return;
      const binding = path.scope.getBinding(localName);
      if (!binding || binding.path.node !== importSpecPath.node) return;
      // …classifyIdentifier as before
    },
    JSXIdentifier(path) { /* same shape */ },
    // Namespaced lookups: visit MemberExpression, verify the object's
    // Identifier binding matches the namespace import specifier.
  });
  return dedupeByPosition(results);
}
```

`classifyIdentifier` stays; it operates on AST parent relationships, which `traverse` exposes via `path.parent` and `path.parentKey`.

### Known edge cases

- **Type-only shadows.** TypeScript lets the same name exist in value and type spaces. `import { User } from …; const User = 1;` — the type `User` still resolves to the import. `traverse`'s scope model tracks them together (Babel doesn't distinguish TS type space). Accept the occasional wrong answer here — rare, and erring toward "reported" is better than silently dropping a type reference.
- **Namespace shadowing.** `import * as ns from …; function f(ns) { ns.symbol; }` — the inner `ns` shadows the import, so `ns.symbol` inside `f` is not a real reference. The binding check catches this (the inner `ns`'s binding is the parameter, not the import).
- **`const { symbol } = …` destructuring.** Creates a new binding named `symbol`; subsequent uses point to the destructured binding, not the import. Existing code already handles this via `classifyIdentifier` returning `null` for `VariableDeclarator.id`, but with scope tracking we can drop that special case and rely on the binding check.
- **JSX.** `<Button />` — Babel's JSX transform gives the JSX identifier the correct binding. `path.scope.getBinding('Button')` from a `JSXIdentifier` path returns the import if not shadowed, the local if shadowed.

### Perf note

A full `traverse` pass is comparable in cost to the existing hand-rolled walker (Babel's traverser is not markedly slower for a single visitor). Cached AST means no double-parse. No scaling concern in 1.7.

## Out of scope (explicit)

- **Symlinks / monorepo TS project references** — treat paths as-is.
- **`export type { X } from …`** distinction in `via` chains — we record it but don't vary the `kind`.
- **Updating `find_symbol` or `get_component_props`** to follow barrels. Separate story if ever needed.
- **New public MCP tools.** 1.7 improves existing answers; shape is additive.

## Future hooks this unlocks

- **Composite `find_and_describe(name)` tool** already has the AST cache; 1.7 adds trustworthy reference counts, which is what makes such a composite's "used 14 times across 8 files" answer meaningful.
- **Rename refactor hint in `find_references`** — "renaming `fetchUser` will break these 14 call sites and require updating these 3 barrel files." The `via` field is the seed for this.
- **Partial dead-code analysis** — an export with zero references (counting barrels + their consumers correctly) is provably dead. Pre-1.7 we couldn't trust this signal because barrels hid references; post-1.7 we can.
